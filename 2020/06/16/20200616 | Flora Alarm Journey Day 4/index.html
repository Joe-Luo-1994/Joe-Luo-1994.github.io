今日透過 QuizApp 學習許多 UI component 及其參數設置，也學習一些在 activity.kt 中設置 listener、修改 UI 參數的方式。

關於 UI Component 部分因用過即知道，暫不花篇幅額外介紹，僅用關鍵字做個備忘。

而除了 Kotlin 部分，今天也試著解決問題：「要做出什麼樣的產品」，會影響到報告、學習、甚至畢業與就業的問題。也會紀錄過程中的收穫。

關鍵字：Intent, library implementation, material design, Cardview, marginStart, 全螢幕效果, object, ScrollView, ImageView, ProgressBar, shape, tools:text, drawable resources, context

Kotlin 部分

以下紀錄查找資料過程中覺得重要的收穫

1. class in Kotlin

因大幅減少了創建 class 的程式碼，除了宣告 class 時省力外，還可用於 function 需 return 超過一個值時的場合，不必再像從前要先回傳 List 再解開。

在下面的演講中，Kotlin Designer 親自說明 Kotlin 與傳統 Java 寫起來應注意的邏輯差異，值得多體會比較，之後打算特地寫一篇紀錄。

ref: How to Kotlin - from the Lead Kotlin Language Designer (Google I/O ‘18)

2. 引入 library 遇到問題時的 sop:

導入 material design 時研究了一下，針對未來不論版本更新或導入新的 library，都可參照以下方法：

  1. Go to Artifact Mappings and serach for desired library
  2. Go to AndroidX Releases page and check for the current version number
  3. Add the library in build.gradoe[module.app] and press Sync Now

ref: android - How to add cardview in layout xml in AndroidX - Stack Overflow

3. drawable resources的 ID

若以 bitmap 方式引入圖片，放在 drawable folder 內的檔案會自動根據檔名生成對應的 property (e.g. R.drawable.bg)，即便看起來像 String 格式，但其實已被轉化為對應 ID 的 Int。

除了目前用過的 Bitmap 之外，似乎還有各種資源的對應方式，遇到時可以看 doc 進一步了解。

產品思維部分

有找到兩個平日可以吸收資訊的網站：

  1. 人人都是產品經理（中國）
  2. 3PM(台灣)

專注技術面時，常會忘了要以需求為導向去做產品，因此透過別人有框架的分析，來完善自己看事物的能力。而如何有效地發現並進一步實現需求則有賴平時的體驗、分析、改良。至於該如何做到？以下整理自己整理的心得

1. 專業分析所需的知識水準要到位

過往像 Extra Credits 試著用娛樂資訊(infotainment)的方式去探討這類需求，但會遇到的最直接瓶頸就是脫離現實，即便是心理層面的探討也都屬於較主觀的觀察。因此包含心理學、統計學、業界的產品分析水準都要精進，才能用更嚴謹的語言去與別人交流產品現況

2. 取得資料與分析資料，再講故事

現實的資料最有說服力，如何透過分析資料讓迭代過程有意義，本身就是可以學習甚至研究的領域。對於資料操作（資料庫）、資料分析（深度學習）的技術面也不可放。

而在那之上如何透過報告或其他機會與人交流，說故事的能力就很重要，因此先從難得的報告機會做起，在那之上看如何擴大影響力。

3. 迭代的技術開發能力學習 - 避開雜訊

Android 是個變動很快的領域，如何更貼近上層開發者的討論框架是我現在的第一要務，避開中間過時或理解錯誤的觀點。

Android 開發論壇應是很棒的材料來源，每天多學多看。而 Udemy 有其他框架思維類的課程，把握時間學習。

心得雜談

今日跟課程不太順利，老師在操作許多 function 時只草草帶過，對於其使用參數的意義沒多做解釋。在對整體架構不清楚的狀況下，看 doc 很辛苦。因此在最短的時間內修完這門課之餘，找時間看架構類的課程來補足觀念
